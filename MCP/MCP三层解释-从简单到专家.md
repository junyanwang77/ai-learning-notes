# MCP 的三层解释：从简单到专家

> 原始对话日期：2026-03-12

---

## 🧒 简单解释（给12岁小孩）

想象你有一个超级聪明的机器人助手（AI），但它被关在一个玻璃房间里——它能看到外面，能说话，但**没办法碰任何东西**。

MCP（Model Context Protocol）就像是给这个机器人装了一双**手**和一套**门钥匙**。

现在它可以：
- 打开你的文件柜，帮你找文件
- 发邮件给你朋友
- 查天气、查日历
- 控制其他程序

**一句话：MCP 是让 AI 从"只会说话"变成"能干活"的标准接口。**

---

## 🕵️ 隐藏的复杂性

表面看起来简单，但藏着很多陷阱：

**1. 信任边界问题**
谁有权限让 AI 执行什么操作？如果恶意网站通过"提示注入"骗 AI 去删文件怎么办？MCP 本身不解决授权，安全责任落在实现者身上。

**2. 状态管理的幻觉**
MCP 的每次工具调用本质上是**无状态的**。"记住上下文"是靠把整段对话历史塞进每次请求——随着对话变长，Token 成本爆炸式增长。

**3. 工具爆炸与选择困难**
当 AI 同时连接几十个 MCP Server（邮件、日历、数据库、代码……），模型要从海量工具中选对一个。工具描述写得不好，AI 就会选错或乱用。

**4. 副作用不可逆**
AI 调用一个"发邮件"工具，邮件就真发出去了。没有天然的"撤销"机制，错误操作代价极高。

**5. 协议本身的局限**
MCP 是 Anthropic 主导的标准，不是中立的 IETF/W3C 标准。生态碎片化风险真实存在，其他厂商可能推出竞争协议。

## 🎓 专家级深度解释

**MCP 的本质是什么？**

MCP（Model Context Protocol）是一个基于 **JSON-RPC 2.0** 的开放协议，定义了 LLM 宿主应用（Host）、客户端（Client）和工具服务器（Server）之间的标准化通信契约。

**架构三角：**
```
┌─────────────┐     MCP Protocol      ┌─────────────┐
│  LLM Host   │ ◄───────────────────► │  MCP Server │
│ (Claude等)  │   JSON-RPC over       │ (工具提供方) │
└─────────────┘   stdio / SSE / HTTP  └─────────────┘
```

**核心原语（Primitives）：**

| 原语 | 方向 | 作用 |
|------|------|------|
| `Tools` | Server→Model | 模型可调用的函数（有副作用）|
| `Resources` | Server→Model | 只读数据源（文件、数据库行）|
| `Prompts` | Server→Model | 预定义的提示模板 |
| `Sampling` | Client→Server | Server 反向请求 LLM 推理 |

**与 Function Calling 的本质区别：**

OpenAI Function Calling 是**模型内协议**，工具定义嵌在 API 请求里，工具执行在客户端。MCP 是**进程间协议**，工具以独立进程运行，通过标准传输层通信，实现了**工具的服务化和复用**。

**传输层设计：**
- `stdio`：本地进程，低延迟，适合 IDE 插件
- `SSE（Server-Sent Events）`：远程服务，支持流式响应
- `Streamable HTTP`（新版）：解决 SSE 的连接保持问题

**最深层的设计哲学：**

MCP 试图解决的核心问题是 **"工具知识的可移植性"**——同一个工具实现，可以被任何支持 MCP 的模型/应用复用，而无需为每个 AI 平台重写集成代码。这本质上是给 AI 生态建立了一个类似 **USB 标准** 的接口层：设备（工具）和电脑（模型）解耦，任意组合。

> 当前最大的工程挑战：在 Agentic 场景下，如何在工具链的长序列调用中保持**意图一致性**，并在出错时实现**优雅回滚**，MCP 协议层面尚无原生答案。
